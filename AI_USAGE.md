## Design Decisions Made by Codex

The prompt only specified seven routes. Codex inferred or chose the following without explicit instruction:

- **Two-factor authentication flow** — the prompt named `/auth/verify-2fa` so Codex designed a full challenge/code/email-log pipeline around it.
- **JWT sessions** — added `jsonwebtoken` and a `sessions` table to back bearer-token auth on the task endpoints.
- **Role-based access** — only admins can create or assign tasks; any authenticated user can view their own tasks.
- **In-process atomic sequence counter** — used for generating 6-digit 2FA codes and unique `login_challenge_id` strings without a random-number dependency.
- **Idempotent SQL schema** — `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` guards so the init script is safe to re-run.
- **`seed_users` conflict guard** — returns HTTP 409 if users already exist, preventing accidental re-seeding.
- **Passwords stored in plain text** — deliberately simplified for the development/assignment context; a production system would use Argon2 or bcrypt.
